# Codebase Audit: code-quality (code obsolète + DRY + refactoring)

Dette de duplication concentrée dans la couche fédération (mappings acteur→modèle et livraison signée réécrits 8-9 fois) et un module `games/front_views.py` monolithique ; peu de code mort, mais plusieurs symboles et 5 templates « composants » orphelins entretiennent des pièges de réutilisation.

- **Date**: 2026-07-19
- **Scope**: `suddenly/**` (19 897 LOC Python, 8 apps) + `templates/**` (175 templates)
- **Health**: fair
- **Findings**: 0 critical, 15 warning, 12 minor

Health `fair` : aucun défaut de correction/sécurité critique, mais duplication systémique réelle dans le cœur fédération (drift silencieux possible) et un monolithe à découper — adressable par lots.

Méthode : `ruff` (F401/F811/F841 = clean), `vulture --min-confidence 60` (fortement filtré des faux positifs Django : admin, signaux, ViewSets `@action`, management commands, middleware, tâches beat), puis vérification manuelle par `grep` croisé (code + templates + tests + config) de chaque candidat. Chaque ligne du tableau cite un `file:line` ouvert et vérifié.

## Top actions (ranked by impact)

Fixes à confier à `aidd-dev:07-refactor` (l'audit n'édite rien). Regroupés pour respecter la Rule of Three du projet.

1. **Unifier le cœur fédération** (résout rows 1, 2, 4, 5, 6) — extraire un helper `actor_model_for`/`content_type_for_actor`, un `sign_and_deliver`, une seule `get_or_create_remote_user` et `_get_or_create_remote_game`. C'est la dette la plus dispersée (9 + 8 sites) et le plus haut risque de drift silencieux sur données fédérées. Effort M, gain élevé.
2. **Purger le code + templates morts** (rows 15-22, 7, 10, 14) — supprimer les 3 factories `create_*_activity`, `build_undo_activity`, `debug_task`, `is_expired`, `LinkRequestStatus.EXPIRED`, les 4 `components/*_card.html` (+ endpoints HTMX fantômes) et `feed/_report_card.html`. Quick wins S ; **décision produit requise** d'abord pour `send_accept_follow`, `is_suddenly_instance`, `EXPIRED`, `build_owned_pc_queryset` (features à câbler ou à retirer — cf. écarts doc↔code, row 15).
3. **Fiabiliser le dispatch inbox** (row 13) — `logger.exception` + persistance/retry des échecs de handler : aujourd'hui un bug de handler renvoie 202 et disparaît. Risque opérationnel fédération.
4. **Factoriser la couche ownership + form-handling** (rows 3, 12) — corriger le helper trompeur `_get_authored_rapport` et centraliser le check auteur ; helper `_render_game_form` partagé create/edit.
5. **Réduire la duplication de templates** (rows 8, 9, 10) — partial `_claim_adopt_fork_actions.html`, généraliser `_actor_avatar.html`, supprimer le doublon `components/quote_card.html`.
6. **Découper `games/front_views.py`** (row 11) — 1 746 l → 4 modules par domaine. Effort L, à planifier séparément une fois la dette de duplication absorbée.
